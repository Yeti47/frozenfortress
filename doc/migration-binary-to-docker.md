# Frozen Fortress — Binary to Docker Migration Guide

This guide walks you through migrating an existing binary-based Frozen Fortress installation to the Docker-native deployment model.

> **Before you start**: create a full backup of your data directory and verify it before proceeding.

---

## Overview

The Docker stack uses the same SQLite database and the same session key files as the binary installation. Migration is essentially about moving your data files into a Docker volume and starting the Compose stack. No database schema changes or data conversion are required.

### What carries over

| Data | Required | Notes |
|------|----------|-------|
| `frozenfortress.db` | Yes | Your entire database — secrets, documents, users, tags |
| `keys/` | Strongly recommended | Persistent session signing/encryption keys generated by Frozen Fortress. Omitting this invalidates active sessions but does not affect encrypted user data. |
| `backups/` | Optional | Existing backup generations; retain if you want continuity |
| `certs/` | Optional | Your existing TLS certificate and key, if you had nginx configured |

### What does not carry over

- **Active sessions**: Redis is treated as ephemeral. All users will need to sign in again after migration.
- **Log files**: Not needed in the new deployment.

## Step-by-Step Migration

### 1. Stop the Binary Installation

Stop the running `ffwebui` process and any associated nginx instance:

```bash
./stop-webui.sh
# or, if running as a systemd service:
sudo systemctl stop frozenfortress
sudo systemctl stop nginx
```

### 2. Back Up Your Data

The default binary data directory on Linux is `~/.config/frozenfortress/`. Create a backup before touching anything:

```bash
cp -r ~/.config/frozenfortress/ ~/frozenfortress-backup-$(date +%Y%m%d)
```

### 3. Download the Docker Release Archive

Download the latest release archive from the [Releases page](https://github.com/Yeti47/frozenfortress/releases) and extract it:

```bash
# Replace <version> with the actual version, e.g. 1.1.0
wget https://github.com/Yeti47/frozenfortress/releases/download/v<version>/frozenfortress-docker-v<version>.zip
unzip frozenfortress-docker-v<version>.zip
cd frozenfortress-docker-v<version>
```

The archive already contains a ready-to-use `compose.yaml` with all images pinned to verified digests, along with `.env.example` and `setup-docker.md`. No repository clone is needed.

### 4. Prepare the Data Volume

The Docker stack expects all application state under a single directory (or named volume) mounted at `/data` inside the `webui` container. The easiest approach is to use bind mounts pointing at your existing data paths.

**Option A — Bind mount the entire directory** (simplest):

Your existing `~/.config/frozenfortress/` maps directly to `/data`. Add bind mounts in a `compose.override.yaml` (so your changes survive `docker compose pull`):

```yaml
services:
  webui:
    volumes:
      - ~/.config/frozenfortress/frozenfortress.db:/data/frozenfortress.db
      - ~/.config/frozenfortress/keys:/data/keys
      - ~/.config/frozenfortress/backups:/data/backups
      - ~/.config/frozenfortress/certs:/data/certs
```

**Option B — Copy to a new host directory**:

```bash
mkdir -p ~/frozenfortress-data/keys
mkdir -p ~/frozenfortress-data/backups
mkdir -p ~/frozenfortress-data/certs

cp ~/.config/frozenfortress/frozenfortress.db ~/frozenfortress-data/
cp -r ~/.config/frozenfortress/keys/         ~/frozenfortress-data/keys/
cp -r ~/.config/frozenfortress/backups/      ~/frozenfortress-data/backups/
# Copy TLS certs if you have them:
cp ~/.config/frozenfortress/certs/frozenfortress.crt ~/frozenfortress-data/certs/
cp ~/.config/frozenfortress/certs/frozenfortress.key ~/frozenfortress-data/certs/
```

Then reference `~/frozenfortress-data` in your volume configuration.

### 5. TLS Certificates

**If you already have a certificate** (from your previous nginx setup): place `frozenfortress.crt` and `frozenfortress.key` in the `certs/` subdirectory of your data volume (see step 4). The nginx container will use them on startup.

**If you did not have a certificate**: leave the `certs/` directory empty. A self-signed certificate will be generated automatically on first startup.

### 6. Start the Docker Stack

```bash
docker compose up -d
```

Watch the startup logs to confirm all services come up cleanly:

```bash
docker compose logs -f
```

### 7. Verify the Migration

1. Open `https://127.0.0.1:8443` in your browser (accept the certificate warning if using a self-signed certificate).
2. Sign in with your existing credentials.
3. Confirm your secrets, documents, tags, and backups are all present.
4. Upload a test image and verify OCR processing starts.

### 8. Decommission the Binary Installation

Once you have confirmed everything works correctly in Docker:

```bash
# Optional: remove the old binaries
rm ~/frozenfortress/bin/ffwebui
rm ~/frozenfortress/bin/ffcli

# Optional: disable any systemd services you created for the binary installation
sudo systemctl disable frozenfortress
```

Keep your backup (`~/frozenfortress-backup-*`) for a while as a safety net.

## Troubleshooting

### Container fails to start / database errors

- Verify that `frozenfortress.db` is readable by the container user. Fix with:
  ```bash
  chmod 644 ~/frozenfortress-data/frozenfortress.db
  ```

### Session keys warning / all users logged out

- If `keys/` was not migrated, Frozen Fortress generates new keys. This is safe — it only means existing sessions are invalidated. Users must sign in again.

### nginx fails with certificate error

- This usually means only one of the two certificate files is present. Either supply both (`frozenfortress.crt` and `frozenfortress.key`) or remove both so a self-signed certificate is generated.

### Checking container logs

```bash
docker compose logs webui
docker compose logs nginx
docker compose logs redis
docker compose logs ollama
```
